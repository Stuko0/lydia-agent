1. ¿Qué es Lydia Agent V2?

Lydia Agent V2 es un agente de IA personal de código abierto construido por Alexnader Villanueva. Ejecuta el mismo core de agente a través de múltiples interfaces: CLI, gateway de mensajería, TUI terminal y aplicación de escritorio Electron. Aprende entre sesiones (memoria + skills), delega a subagentes, ejecuta tareas programadas y controla un terminal y navegador reales. Se extiende principalmente mediante plugins y skills, no haciendo crecer el core.

Líneas de código analizadas: más de 50,000 líneas Python en el árbol de código, distribuidas en 141 módulos del agent core, 195 módulos CLI, 108 herramientas, 67 archivos gateway y 175 plugins. La versión actual es 0.17.0.

1.1 Propiedades de diseño fundamentales

Propiedad Impacto
Caché de prompt por conversación Cada conversación larga reutiliza un prefijo en caché. Mutar el contexto pasado, cambiar toolsets o reconstruir el system prompt invalida el caché y multiplica el costo del usuario.
Core como narrow waist Cada core tool se envía en cada llamada API. La barrera para un nuevo core tool es altísima. La mayoría de nueva capacidad debe llegar como CLI command + skill, plugin o MCP.
Config vs. Secrets .env solo para credenciales (API keys, tokens, passwords). Toda configuración de comportamiento (timeouts, thresholds, feature flags) va en config.yaml.

2. Arquitectura General del Agente

El sistema sigue un patrón de waist design: un core de agente estrecho y reutilizable que se conecta a múltiples interfaces (CLI, TUI, gateway) y se extienda radialmente a través de plugins y skills. El core nunca debe hacer crecer su superficie: cada tool core se envía en cada llamada API, por lo que cada nueva herramienta tiene un costo permanente multiplicado por el uso.

2.1 Flujo de ejecución principal (CLI/TUI)

La entrada del usuario viaja por run_agent.py (5,709 líneas) → agent/server.py → agent/chat_completion_helpers.py → modelo → herramienta o respuesta. Cuando el agente necesita ejecutar una acción (terminal, file, browser, delegate), el core genera un tool call que el runtime ejecuta de forma segura y devuelve el resultado al mismo mensaje de la conversación.

Fase Componente Descripción
1. Entrada lydia_cli/main.py (13,668 líneas) o gateway/run.py (19,114 líneas) Lee el mensaje del usuario desde CLI, TUI o plataforma (Telegram, Discord, etc.)
2. Preparación de contexto agent/context_manager.py Construye el contexto: memory (FTS5), skills hub, session search, project context, cron jobs
3. Llamada al modelo run_agent.py → model_tools.py → providers/ OpenAI-compatible API call. Soporta Nous Portal, OpenRouter, OpenAI, custom, y 6+ backends
4. Respuesta run_agent.py (parse, tool calls) o agent/server.py (streaming) Si hay tool calls, se ejecutan y se reintegran; si no, se entrega la respuesta al usuario
5. Ejecución de herramientas tools/ (108 archivos, 7.3 MB) Cada tool se ejecuta en contexto aislado con timeout, límites de output y sandboxing
6. Feedback loop run_agent.py + lydia_state.py (5,659 líneas) El resultado de la herramienta se envía al modelo como mensaje de sistema, loop de tool-call

3. Arquitectura de Gateway

El Gateway es el daemon de mensajería multi-plataforma. Su módulo principal, gateway/run.py (19,114 líneas), gestiona la conexión con clientes de mensajería externa, procesa mensajes entrantes, los envía al core del agente para su procesamiento y devuelve las respuestas a la plataforma apropiada.

3.1 Componentes del Gateway

Componente Responsabilidad
gateway/run.py (19,114 líneas) Daemon principal. Inicia el bucle de eventos, maneja la reconexión y las notificaciones de herramientas.
platforms/ Adaptadores de plataforma para diferentes servicios de mensajería (Telegram, Discord, Slack, WhatsApp, Signal, etc.). Cada plataforma tiene su propio adaptador.
config.yaml Configuración del gateway, incluidos tokens de API, ajustes de plataforma y opciones de autenticación.

3.2 Integración con la Plataforma

Cada plataforma requiere sus propios endpoints de API y configuración de seguridad. Las configuraciones típicas incluyen tokens de API o certificados OAuth, ajustes de WebSocket, o endpoints webhook para la entrega asincrónica de mensajes.

3.3 Funcionalidades Clave del Gateway

  • Soporte Multi-plataforma: Conecta con varios servicios de mensajería a través de adaptadores. El soporte para plataformas puede estar en desarrollo activo.
  • Procesamiento de Mensajes: Gestiona la entrega de mensajes entrantes y salientes, las respuestas automatizadas y el enrutamiento de comandos.
  • Gestión de Sesión Persistente: Mantiene el contexto de la conversación y el estado entre las interacciones.
  • Escalado: El daemon centralizado permite el acceso multi-sesión a través de múltiples canales con un único motor de agente.
Nota: La integración con cada plataforma puede requerir ajustes específicos de API y configuraciones de seguridad. La compatibilidad con algunas plataformas puede no estar disponible completamente.

4.2 Configuración del Sistema (config.yaml)

config.yaml es el archivo de configuración principal. Sigue la regla: .env solo para secrets (API keys, tokens) y config.yaml para toda la configuración de comportamiento (timeouts, thresholds, feature flags, display prefs).

Sección Propósito Ejemplos
servers Configuración de proveedores de modelos LLM base_url, api_key (en .env), model name, temperature, max_tokens
gateway Opciones de gateway de mensajería telegram_token, discord_bot_token, webhook_url
tools Configuración de herramientas habilitadas terminal.enabled, browser.enabled, file.enabled, delegation.enabled
memory Opciones de memoria persistente auto_save_interval, fuzzy_threshold, fts5_enable
tools Herramientas habilitadas y configuradas {enable, disable, enable_all, disable_all}
models Catálogo de modelos disponibles {list, switch, set_cost}
skills Gestión de habilidades {list, create, update, search}
plugins Instalación y gestión de plugins {list, install, enable, disable}
cron Jobs programados {list, create, delete, run, update}
Nota: Todos los valores de configuración conductual van en config.yaml, nunca en .env. Solo las credenciales (API keys, tokens, passwords) se almacenan en .env.

5.3 Footprint Ladder (Jerarquía de Implementación)

Para evitar inflar el core, Lydia sigue esta jerarquía al añadir nueva capacidad:

  1. Extender código existente — la opción más barata
  2. CLI command + skill — nueva funcionalidad empaquetada como skill
  3. Service-gated tool (check_fn) — habilitada por chequeo dinámico
  4. Plugin — extiende el core sin tocarlo
  5. MCP server — servidor externo via Model Context Protocol
  6. Core tool nueva — último recurso, requiere justificación

6. Memoria y Aprendizaje Continuo

Lydia es el único agente con un bucle de aprendizaje incorporado: crea skills desde la experiencia, las mejora durante el uso, persiste conocimiento entre sesiones, busca sus propias conversaciones pasadas y construye un modelo profundo del usuario.

Session Search SQLite FTS5 Búsqueda de conversaciones pasadas: recupera sesiones por contenido, scroll dentro de una sesión, resolución de referencias @session:/.
Skills (memoria procedural) SKILL.md + YAML frontmatter Procedimientos reutilizables: workflows, comandos, pitfalls. Se crean desde la experiencia y se actualizan automáticamente.
User Profile Perfil estructurado Quién es el usuario: nombre, rol, preferencias, estilo de comunicación. Se construye y refina con cada interacción.
Context Engine Plugin + agent context Preparación de contexto enriquecido antes de cada turno: memoria + skills + proyectos + últimas sesiones + cron jobs.

7. Cron Jobs y Tareas Programadas

El sistema de cron (7 módulos, ~485 KB) permite programar tareas que el agente ejecuta automáticamente en segundo plano. Las tareas corren en sesiones aisladas y entregan resultados al usuario sin intervención manual.

7.1 Componentes del Sistema Cron

MóduloFunción
cron/scheduler.pyPlanificador principal: evalúa schedules, gestiona ejecuciones
cron/jobs.pyGestión de jobs: crear, listar, actualizar, pausar, eliminar
cron/blueprint_catalog.pyCatálogo de blueprints predefinidos para jobs comunes
cron/suggestion_catalog.pySugerencias inteligentes basadas en el uso del usuario
cron/suggestions.pyGenerador de sugerencias para automatización

7.2 Capacidades del Cron

  • Schedules flexibles: cada 30m, cada 2h, cron expression, ISO timestamp
  • Script mode: jobs que ejecutan scripts directamente sin LLM (watchdog pattern)
  • Agent mode: jobs que ejecutan prompts con el agente completo
  • Chained jobs: un job puede consumir el output de otro (context_from)
  • Multi-entrega: Telegram, Discord, Slack, local, o todas las plataformas
  • Auto-contexto: los jobs heredan AGENTS.md, .cursorrules del workdir
